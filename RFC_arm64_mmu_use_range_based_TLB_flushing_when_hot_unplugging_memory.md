# [RFC] arm64: mmu: use range based TLB flushing when hot unplugging memory

---

## 更新 - 2026-05-21 09:50 UTC

## 核心话题
该邮件讨论围绕一个针对 ARM64 架构的 RFC 补丁展开，补丁的作者 Alistair Popple 试图解决内存热拔出时 TLB 无效化性能极差的问题。当前 ARM64 在热拔出内存时，需要逐页解除直接映射（direct map）然后进行 TLB 刷新，无法利用基于范围的 TLB 刷新（range-based TLB flushing），导致清除大容量内存耗时过长甚至触发 RCU 停滞警告。Alistair 给出的数据是：拔掉 480GB 内存原需约 1 分钟，应用补丁后降至约 1 秒，性能提升 60 倍。该补丁的修改位于 `arch/arm64/mm/mmu.c`，仅调整了 TLB 刷新的时机，试图在解除映射后一次性对整段地址范围做 TLB 刷新。

Alistair 将补丁标记为 RFC 是因为他不确定修改是否安全：补丁在释放 PTE 页面之后才执行 TLB 刷新。他担心 ARM64 硬件可能通过过时的 TLB 条目修改页面内的访问 / 脏标志位，从而在页面已被释放但仍被写入的窗口期造成危险。他提出一种更安全的方案：先将所有待释放页面收集到链表中，执行范围 TLB 刷新后再释放，并希望先获得对整体思路的反馈。

Ryan Roberts（Arm）回复指出，该补丁无法合入 v7.1-rc4，因为与已合入主线的一个提交冲突：`48478b9f79137 ("arm64/mm: Enable batched TLB flush in unmap_hotplug_range()")`。该提交也带来了类似的性能改进，Ryan 询问这是否已经解决了 Alistair 的问题。Ryan 进一步解释了利用同样逻辑的两条路径：解除线性映射（linear map）与解除对应的 vmemmap。对于 vmemmap 路径，由于内存随后被释放，必须在释放前刷新 TLB，因此无法安全地使用范围优化；但线性映射路径正是最慢、最受影响的部分，所以现有补丁可能已足够满足需求。

## 达成的结论
本次讨论未形成新的一致结论。Ryan 仅仅是告知 Alistair 已有类似的优化补丁合入，并推测该补丁可能已解决性能问题，同时指出了 vmemmap 释放路径中范围 TLB 刷新的固有限制。Alistair 尚未作出进一步回应，因此关于 Alistair 原始 RFC 是否还需要额外改进，以及如何处理 vmemmap 释放场景的安全性问题，讨论处于开放状态。

## 下一步改进方向
1. **验证现有补丁的效果**：Alistair 需要基于已包含 `48478b9f79137` 的内核版本重新测试 480GB 内存热拔出的耗时，确认批量 TLB 刷新是否已经达到可接受的性能水平。
2. **评估 vmemmap 路径的优化必要性**：如果线性映射的批量刷新已解决主要瓶颈，可能无需为 vmemmap 路径引入复杂且危险的释放前收集机制。如仍存在不可接受的延迟，则需设计安全方案，例如先收集所有要释放的页面，执行全范围 TLB 刷新后再批量释放。
3. **讨论硬件行为细节**：针对 ARM64 硬件能否通过过时 TLB 条目异步修改页表位（如访问 / 脏位），可征询更广泛的架构专家意见，以确定安全窗口是否存在。
4. **更新补丁或放弃**：Alistair 可根据反馈要么撤回当前 RFC，要么在此基础上开发新版本以处理 vmemmap 情况。

## 新增补丁
在该邮件线程中，除了原始 RFC 补丁（由 Alistair Popple 发布）外，未出现任何新的补丁版本。Ryan 的回复仅提及了已合入主线的提交 `48478b9f79137`，并未附带新补丁。
