# KVM: arm64: Handle PKVM_HYP_REQ_HYP_ALLOC request

---

## 更新 - 2026-05-20 16:26 UTC

## 核心话题
该补丁为 ARM64 pKVM（受保护 KVM）子系统引入了一个新的 hypervisor 请求类型 `PKVM_HYP_REQ_HYP_ALLOC`，其核心目的是让运行在异常级别 EL2 的受保护虚拟化管理程序能够在自身堆内存（heap allocator）不足时，主动向运行在 EL1 的主机（host）请求补充可用页面。

补丁首先在 `pkvm_hyp_req_type` 枚举中增加了 `PKVM_HYP_REQ_HYP_ALLOC` 条目，紧接着调整了辅助函数 `pkvm_hyp_req_arg_size()`，使得对于该请求类型返回 `sizeof(req->mem)` 作为参数区域大小，确保主机能够正确解析请求中携带的内存需求信息。最为关键的是，在主机端的请求处理函数 `pkvm_handle_hyp_req()` 中添加了对新类型的处理分支：当主机收到此请求时，调用 `pkvm_hyp_topup(PKVM_TOPUP_HYP_ALLOC, req->mem.nr_pages)` 来向管理程序的堆分配器“充值”指定数量的页面。

这个机制的技术动机在于 pKVM 的设计要求将受保护虚拟机的执行隔离到 EL2 管理程序中，并严格限制主机操作系统对虚拟机内存和状态的访问。管理程序本身在初始化阶段从主机分配了一定量的内存用作自身的堆，但随着受保护虚拟机数量的增加或运行时动态内存需求的增长（例如创建页表、存储虚拟化状态等），预先分配的堆可能被耗尽。为了让管理程序能够动态扩展工作内存而不破坏安全隔离模型，需要一种受控的方式由主机提供额外页面。本补丁正是提供这样一种受控通信信道：管理程序通过共享内存接口发送 `PKVM_HYP_REQ_HYP_ALLOC` 请求，主机安全地挂起其他操作来处理该请求，将页面通过 `pkvm_hyp_topup` 安全地映射并交给管理程序，从而确保管理程序不会因为内存不足而崩溃或拒绝服务。

该补丁是系列化工作的一环，从编号 `[PATCH 08/17]` 可以看出它属于一个 17 个补丁的多步实现，很可能在前面已经定义了 `pkvm_hyp_topup` 和 `PKVM_TOPUP_HYP_ALLOC`，并为堆分配器预留了充值接口。补丁本身代码量很小，但触发了 pKVM 弹性资源管理的关键路径。

## 下一步改进方向
该补丁将等待社区维护者和相关开发人员的审核。可能需要关注以下几个方面：`PKVM_HYP_REQ_HYP_ALLOC` 请求的错误处理是否充分（例如当主机无法满足充值请求时的行为）；与系列中其他补丁的接口一致性；安全性方面是否需要限制可请求的页面数量以防止主机资源被恶意耗尽；以及是否需要增加追踪点或调试信息来监控该关键分配路径。待反馈后可能需要对枚举值定义、参数大小计算或充值回调进行修正。
