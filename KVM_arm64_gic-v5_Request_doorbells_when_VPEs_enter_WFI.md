# KVM: arm64: gic-v5: Request doorbells when VPEs enter WFI

---

## 更新 - 2026-05-21 14:54 UTC

## 核心话题
本邮件是 Linux 内核 ARM64 KVM 虚拟化系列补丁（v2 版本共 39 个补丁中的第 16 个）的一部分，主要解决 GICv5（即 ARM 通用中断控制器 v5，带有 ITS 支持）虚拟化场景下 VPE（Virtual Processing Element）进入 WFI 状态时的唤醒机制问题。当客户机 vCPU 执行 WFI 指令后，KVM 会将其对应的 VPE 标记为 non-resident（非驻留），以便让出物理 CPU 资源。此时，如果有新的 SPI（共享外设中断）或 LPI（局部外设中断）为该 VPE 挂起，硬件需要一种通知 KVM 的方式，以便重新调度该 vCPU。补丁引入了“VPE doorbell”机制：在 VPE 变为 non-resident 之前，KVM 向 GICv5 请求一个门铃中断，并配置其优先掩码（DBPM）为当前虚拟 CPU 的有效虚拟优先掩码。这样硬件仅在中断优先级满足客户机当前掩码（即中断可以被客户机接受）时才触发门铃，避免因被客户机屏蔽的中断而无谓唤醒 vCPU。补丁的关键技术点包括：
- **DBPM 编程**：将虚拟优先掩码写入 doorbell 优先级掩码字段，使硬件过滤掉不能实际投递的中断。
- **门铃亲和性设置**：在请求门铃前，将其绑定到当前物理 CPU，即最后运行该 vCPU 的 CPU，以利用 Cache 热度、减少唤醒延迟，并同时让门铃中断在不同物理 CPU 间分散，避免集中在单一 CPU 上。
- **清除 db_fired 状态**：在 VPE 变为 non-resident 前，先清除之前的 db_fired 标志，确保能够正确捕捉到新的门铃触发。
- **虚拟中断挂起检查扩展**：修改 `kvm_vgic_vcpu_pending_irq()` 函数，使其在检查 vCPU 是否有待处理中断时，不仅考虑已有的 PPI 挂起，还检查 VPE doorbell 是否已触发，从而让 KVM 能及时识别需要被唤醒的 vCPU。

邮件中的代码片段（虽然被截断）显示了 `__vgic_v5_make_resident` 和 `vgic_v5_` 相关函数，体现了这些逻辑在 VPE 状态切换路径中的集成。该补丁明确针对 GICv5（支持 ITS 的 GIC）而设计，是 KVM 虚拟化性能优化的重要一环，尤其适用于需要高效处理稀疏 LPI 的 ARM 服务器平台。
