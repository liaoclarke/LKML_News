# perf/arm-cmn: Add workarounds for CMN-S3 on Graviton5

---

## 更新 - 2026-05-21 17:02 UTC

## 核心话题
该邮件讨论围绕为 Amazon Graviton5 服务器平台上使用的自定义 CMN‑S3（ARM 一致性网格网络第三代）硬件提供 perf 驱动支持而展开。Graviton5 上的 CMN‑S3 实现存在特殊行为：部分硬件发现寄存器中本应包含有效信息的字段被硬件清零，导致标准 ARM CMN 驱动无法正确获取 DTC（DynamIQ 拓扑控制器）的域信息和逻辑 ID，从而无法正常初始化性能监控单元（PMU）。

提交者 Aviv Bakal 提供了 v2 版本的补丁，旨在通过以下方式绕开硬件瑕疵：
1. **引入专用 ACPI HID**：使用新的硬件 ID 标识 Graviton5 的 CMN 变体（代码中定义为 `PART_GRAVITON5 = 0xa5`，并在发现阶段覆盖为 `PART_CMN_S3`），以便在驱动初始化时能够识别该特定实现并采用修正逻辑。
2. **推导 DTC 域**：既然单元信息寄存器中的 DTC 域字段被零化，就从 XP（交叉点）节点 ID 中按特定规则反推出真实的 DTC 域。补丁中出现了 `arm_cmn_graviton5_dtc_domain()` 函数，利用 XP ID 的位操作（如 `(xp_id >> 7)`）来计算出域编号。
3. **设置 DTC 逻辑 ID**：节点信息寄存器中 DTC 逻辑 ID 字段同样被零化，因此 v2 改为使用前面计算得到的域 ID（`xp->dtc`）直接赋值给 DTC 的逻辑 ID，而不是之前 v1 中尝试使用 XP 自身的逻辑 ID。这一变更是对上一版评审意见的回应。
4. **扩展最大网格维度**：因为 Graviton5 的实际网格尺寸超过了 CMN‑S3 架构允许的最大值 12，补丁将 `CMN_MAX_DIMENSION` 从 12 提升至 14，以容纳 14×14 甚至更大的拓扑。

Robin Murphy（ARM 架构维护者）在邮件中回复，但由于邮件被截断，我们无法看到其完整的评审意见。一般来说，此类讨论会聚焦于变体识别策略（如是否应使用独立的 ACPI HID，或通过其他寄存器特征来激活工作区）、域推导算法的可靠性和可维护性，以及驱动结构是否需要更抽象的修正钩子来应对零值字段问题。从 v2 已作出的修改来看，前一阶段已经接受了“使用计算域 ID”的改进方向，因此 v2 主要解决细节实现问题。

## 下一步改进方向
1. **补充 Robin Murphy 的评审意见**：需根据 Robin 对 v2 的完整回复决定是否需要进一步改动。可能的关注点包括：
   - `PART_GRAVITON5` 这种合成部件号的定义方式是否合理，是否有可能与其他供应商 ID 冲突。
   - `arm_cmn_graviton5_dtc_domain()` 中利用 XP ID 位域的方式是否足以覆盖所有可能的 Graviton5 拓扑，有无边界情况。
   - 将 `CMN_MAX_DIMENSION` 全局提升至 14 是否会影响其他平台的资源分配（如数组大小），是否应改为动态分配或使用平台特定的最大值。
   - 将域推导逻辑硬编码为平台特定函数的做法，能否抽象成更通用的“零字段修复”流程，以便未来有类似硬件异常时复用。
2. **相关测试**：在 Graviton5 实际硬件上进行功能测试，确保 perf 的 CMN PMU 事件计数、通道发现和拓扑解析均正确；同时也需要在其他 CMN‑S3 平台上执行回归测试，确认新增的 ACPI HID 和维度变更不引入副作用。
3. **文档/ACPI 表配合**：内核侧补丁就绪后，需确保 Graviton5 固件（BIOS/UEFI）提供正确的 ACPI HID，以便驱动能够自动匹配到工作区代码。
