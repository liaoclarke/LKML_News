# arm64: dts: qcom: glymur: Add GPU cooling

---

## 更新 - 2026-05-22 15:42 UTC

## 核心话题
该邮件讨论的是为高通 Glymur 平台（一种 ARM64 架构的移动平台）的设备树增加 GPU 主动冷却（thermal throttling）支持。补丁作者指出，当前该平台的 GPU 在温度过高时并不会自动降低频率，缺乏过热保护机制，可能导致硬件损坏或系统不稳定。因此，补丁引入了一套基于温度传感器（tsens）的被动冷却方案：当 GPU 温度达到 95°C 时，通过冷却映射将 GPU 设备的运行频率限制在较低水平，以降低发热，直至温度下降到安全范围。

补丁主要修改了 `arch/arm64/boot/dts/qcom/glymur.dtsi`，内容涉及以下几个关键点：
- 为多个 GPU 温度传感器对应的热区域（thermal zone）增加了 `polling-delay-passive = <100>`，即被动轮询间隔设为 100 毫秒，确保温度监控及时。
- 将原有仅配置为“hot”类型的静态跳变点（90°C）改为“passive”类型的主动跳变点（95°C），并降低了迟滞值（hysteresis）从 5000 到 1000，使得温度控制更加灵敏。
- 在每个热区域中新增了 `cooling-maps` 节点，指定当温度达到 `gpuXX_alert0` 跳变点时，触发对 GPU 设备的冷却动作，冷却状态上下限均设为 `THERMAL_NO_LIMIT`，表示允许冷却框架自由选择最合适的运行频率（通常是最低频率）。
- 补丁还包含了必要的头文件包含（`<dt-bindings/thermal/thermal.h>`）以支持上述常量。

从技术角度看，该方案利用了 Linux 内核通用热管理框架（thermal framework）的被动冷却机制，将温度传感器数据与设备运行频率动态关联。通过将跳变点类型改为“passive”，内核会在温度超过 95°C 时激活冷却设备（GPU），并依据 cooling state 梯队降低 GPU 频率，直到温度恢复。使用 `THERMAL_NO_LIMIT` 表示不限制冷却行为的最大或最小状态，交由冷却设备自身的频率表（OPP table）和步进逻辑处理，这是一种常见的简化配置，适用于只有单一冷却行为（降频）的设备。

补丁中涉及多个 GPU 热区域（如 `gpu-0-0-thermal`、`gpu-0-1-thermal` 等），表明该平台有多路 GPU 温度传感器，补丁对每一路都做了相同处理，确保全面覆盖。邮件中强调“GPU does not throttle its speed automatically”，说明这原来是平台安全特性的缺失，该补丁正是为了填补这个空白，提升热管理鲁棒性。
