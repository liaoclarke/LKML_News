# KVM: arm64: Add pkvm_hyp_req infrastructure

---

## 更新 - 2026-05-20 16:26 UTC

## 核心话题
该邮件是 pKVM（protected KVM）补丁系列中的第 07/17 号补丁，由 Vincent Donnefort 提交。其核心目标是**为 pKVM 虚拟机监控器（hypervisor）向主机（host）请求资源的能力构建基础架构**。在 ARM64 的 pKVM 模型中，一部分代码运行在保护执行环境中，无法直接访问主机的内存管理或 I/O 资源。当 hypervisor 需要动态获取内存页面、映射地址空间等操作时，就必须通过一条清晰的“请求—响应”通道向主机发起调用，并在主机完成处理后重试之前的异常（HVC）。

该补丁引入的核心结构体 `struct pkvm_hyp_req` 封装了一次请求的类型和参数。当前定义的枚举类型 `pkvm_hyp_req_type` 仅包含 `PKVM_HYP_NO_REQ = 0`，表示无请求，以及 `__PKVM_HYP_REQ_TYPE_MAX` 作为边界占位符。请求参数通过联合体（union）承载，其中 `mem` 分支包含 32 位的 `nr_pages`，暗示后续会使用该通道请求分配连续的物理内存；`args` 分支则是一个字节数组，大小由 `PKVM_HYP_REQ_SMCCC_ARG_SIZE_MAX` 宏限定，以便将请求参数直接序列化到 SMCCC 调用的寄存器 a2 及之后的位置。

补丁提供了两个关键的内联辅助函数：`pkvm_hyp_req_to_smccc()` 负责将 pending 请求的类型和参数编码放入主机上下文（`host_ctxt->regs.regs[2..]`），使得 hypervisor 能够通过 SMCCC 机制向主机传递信息；对应的解码函数 `smccc_to_pkvm_hyp_req()`（虽在截断片段中未完全展开）用于主机制析请求。宏 `pkvm_call_hyp_req()` 则完成了自动化闭环：它在调用 HVC 后检查返回结果中是否携带挂起的请求，若存在则调用主机端注册的处理函数，处理完毕后重新执行 HVC，从而实现了“请求-处理-重试”的无缝循环。

此外，补丁还引入了 trace event 以跟踪请求处理过程，便于调试和性能分析。该补丁是整个 pKVM 动态资源管理能力的基石，为后续引入具体的内存捐赠、页面回收等机制铺平了道路。

## 下一步改进方向
- 该补丁需要经过社区（尤其是 KVM/ARM64 维护者）的 review，重点关注请求通道的安全性和正确性。
- 当前 `pkvm_hyp_req_type` 枚举只定义了占位值，**需要随系列后续补丁补全实际请求类型**，例如页面分配、IO 映射等，并实现对应的 `pkvm_hyp_req_arg_size()` 分支。
- 解码函数 `smccc_to_pkvm_hyp_req()` 的实现需在后续代码中完整给出，并确保其与编码函数的严格对齐。
- 宏 `pkvm_call_hyp_req()` 依赖主机端注册的请求处理句柄，需补充相应注册接口和实际处理逻辑（可能出现在系列的其他补丁中）。
- 需完善 trace 事件的使用点，并在文档或 cover letter 中说明整个请求机制的协议设计和错误处理策略。
- 测试覆盖：需在 pKVM 环境下验证请求触发与重试流程，确保在内存压力或并发场景下的稳定性。
