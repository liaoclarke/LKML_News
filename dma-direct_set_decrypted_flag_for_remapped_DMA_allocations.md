# dma-direct: set decrypted flag for remapped DMA allocations

---

## 更新 - 2026-05-22 09:58 UTC

## 核心话题
该邮件展示的是针对 ARM64 架构（及机密计算扩展）的 DMA 直接分配器补丁，源自 `dma-direct.c` 文件的修复。补丁的核心问题是：当设备属于 DMA 非一致性（non-coherent）且需要重映射（remap）时，原有的代码路径在分配时遗漏了对 `dma_set_decrypted()` 的调用，导致 DMA 缓冲区在内存中仍然保持加密状态——即使设备要求以明文（非加密）方式访问内存。这对于支持机密计算的架构（如 AMD SEV、ARM CCA 等）至关重要，因为共享的 DMA 缓冲区必须被解密才能被设备正确访问，否则会出现数据错乱或访问错误。

邮件作者 Aneesh Kumar 指出，之前提交中引入的漏洞（由 `f3c962226dbe` 引入）导致重映射路径跳过了解密标记，所以本次修复重新调整了代码分支位置，使得无论是直接映射还是重映射，都能在分配后、使用前调用解密函数，若解密失败则干净地释放内存。补丁还做了两处关键改动：一是将 `set_memory_*()` 操作的对象从重映射后的 CPU 虚拟地址改为后备物理页的线性映射别名（linear-map alias），这样才能真正影响到内核线性映射区域，保证 Coherent 访问的可见性；二是明确禁止在带有 `DMA_ATTR_CC_SHARED` 属性时分配 highmem 页面，因为 highmem 页面没有线性映射，无法进行页面属性的设置和查找 PFN 以施加架构要求的 PFN 属性，这对于机密共享 DMA 是必需的。

补丁的变更集中在对 `dma_direct_alloc()` 的修改：新增布尔变量 `allow_highmem`，并在检测到 `DMA_ATTR_CC_SHARED` 时将其设为 `false`，避免分配 highmem 页；调整 `dma_set_decrypted()` 的调用位置到重映射分支之后；修改 `dma_direct_free()` 中的解密恢复操作，确保同样针对线性映射地址进行。整个修改旨在正确支持具有加密/解密需求的 DMA 设备，确保 DMA 缓冲区的机密隔离与共享的正确性。
