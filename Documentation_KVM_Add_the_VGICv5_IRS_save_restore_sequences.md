# Documentation: KVM: Add the VGICv5 IRS save/restore sequences

---

## 更新 - 2026-05-21 15:02 UTC

## 核心话题
本邮件是 ARM 架构 GICv5 虚拟化支持系列补丁的最后一篇，主要内容是为 KVM 的 VGICv5 IRS（Interrupt Redistribution Service）状态保存与恢复流程添加文档说明。作者强调，若违背正确的保存/恢复顺序，“几乎必然导致无法正确恢复能够处理中断的客户机”，这突显了该文档所描述的顺序对于确保客户机中断子系统在迁移后能继续正常工作的关键性。

技术要点方面，补丁严格规定了保存与恢复两个方向的依赖关系。保存时，IST（Interrupt Storage Table）必须**先于**客户机内存保存，因为客户机的 LPI IST 内容是由 VGIC 写入客户机内存的。恢复时顺序相反，必须**先恢复客户机内存**，然后才恢复 IST，这样恢复 IST 时才能从已复原的内存中正确读出 LPI 配置。

在恢复 IRS 自身的 MMIO 寄存器时，顺序同样至关重要。首先恢复 `IRS_IDx` 寄存器，因为它们定义了 IRS 的能力边界，是后续创建和管理 IST 及 SPI 的基础。接着恢复 `IRS_IST_CFGR`，然后才是 `IRS_IST_BASER`。这一顺序确保了 KVM 在处理 `KVM_DEV_ARM_VGIC_GRP_IST` 恢复操作时，能依据已恢复的配置判断客户机 LPI IST 是否存在、所需大小以及客户机提供的迁移储存位置，从而正确分配并填充宿主机侧的 LPI IST。

补丁还指出，剩余的 MMIO 寄存器（如 SPI 控制相关寄存器）应在此之后恢复。其中 SPI IST 从用户态提供的缓冲区中提取并传输到宿主机已分配的 SPI IST；LPI IST 则从客户机内存中抽取并写入宿主机分配的 LPI IST。作为一般原则，`IRS_*_STATUSR` 状态寄存器在恢复时可以忽略，因为它们不可由用户空间写入。该文档为用户态 VMM（如 QEMU）完成正确的迁移序列提供了权威参考。
