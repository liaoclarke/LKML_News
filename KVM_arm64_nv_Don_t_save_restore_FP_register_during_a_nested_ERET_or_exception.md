# KVM: arm64: nv: Don't save/restore FP register during a nested ERET or exception

---

## 更新 - 2026-05-21 07:35 UTC

## 核心话题
该讨论围绕 KVM/arm64 嵌套虚拟化（NV）场景下的 FPSIMD/SVE 状态管理优化展开。Marc Zyngier 提出了 v2 补丁系列的第二部分，目的在于消除嵌套 ERET 或异常交付时对 FP 寄存器的不必要保存/恢复操作。当前方案中，在 L1 与 L2 之间切换时，会通过 `kvm_arch_vcpu_put()` 保存旧状态、修改内存中的状态，再通过 `kvm_arch_vcpu_load()` 加载新状态。这会导致所有活跃的 FPSIMD/SVE 状态被保存并从 CPU 上解绑，待后续陷入时再延迟恢复。然而，FPSIMD/SVE 状态在不同异常级别之间是共享的，切换时只需更新少数控制寄存器（陷阱控制和向量长度设置），无需完整保存/恢复共享的寄存器文件。随着嵌套深度增加，这种不必要的陷阱开销呈指数级增长，严重影响 I/O 密集型负载的性能。

Marc 的补丁通过利用 vcpu 标志来识别 nested ERET 或异常注入上下文，完全跳过 `kvm_arch_vcpu_put()` 和 `kvm_arch_vcpu_load()` 中的 FP 保存/恢复过程。共享的 FPSIMD/SVE 状态直接保留在 CPU 上，仅根据 EL 级别和嵌套上下文重新计算有效的陷阱控制值（在 `__activate_traps()` 中）以及向量长度设置（在 `fpsimd_lazy_switch_to_guest()` 中）。性能评估显示，对于 EL1 L3 客户机、L1 和 L2 均启用该优化且产生大量嵌套异常的高 I/O 负载，可获得至少 10% 的墙钟时间缩减。

Joey Gouly 通过提问确认了优化的原理：L1 将 L2 的 FP 状态设置到 CPU 后执行 ERET，该 ERET 陷入到 L0，此时关闭抢占，但优化后的路径会避免 `put_fp()`/`load_fp()`，因此 L2 直接看到 L1 留在硬件寄存器中的 FP 状态，随后通过设置 CPTR 陷阱使得 L2 首次访问 FP 时延迟恢复。Marc 肯定了这一理解，并补充 L1 到 L2 的异常注入路径也具有类似形态。

Mark Rutland 则指出了 `kvm_arch_vcpu_put()` 和 `kvm_arch_vcpu_load()` 之间窗口期可能存在的安全隐患（邮件截断，具体风险未完全展示）。这暗示在跳过保存时，该窗口期内共享状态可能被不当修改或错乱，需要进一步审查。

## 达成的结论
目前尚未形成最终共识。Marc 和 Joey 确认了补丁的核心逻辑和性能收益，但 Mark Rutland 提出的关于 `put()` 和 `load()` 之间窗口期的安全考量尚未得到答复（邮件截断）。因此，讨论可能仍在进行中，后续需要针对该窗口期问题给出明确解释或代码调整。

## 下一步改进方向
1. 回应并解决 Mark Rutland 提出的窗口期安全担忧，可能需要增加必要的同步机制或证明该窗口期内状态不会被其他路径破坏。
2. 审查是否需要为 SME 扩展预留类似的优化框架，待 SME 支持时一并处理。
3. 可能需要对嵌套不同层级、不同陷阱路径进行更全面的测试，以确保无寄存器泄露或状态腐败。
4. 若窗口期问题无实质性风险，应补充文档或注释说明该优化为何安全。
